# 通用 AI 协作工作流（Generic Workflow Template）

## 概述

本模板提炼自 design / rule-authoring / verification / release-packaging 四个特化 skill 的共性流程，定义了 AI 执行任何任务时必须遵循的 **6 阶段标准流程**。特化 skill 在此基础上叠加领域约束即可。

## 核心理念

| 原则 | 含义 |
|------|------|
| **反 slop** | AI 先复述理解，用户来纠——而非用户来确认 |
| **外部真相源** | 决策落文件（spec），不依赖会腐烂的对话上下文 |
| **Checkpoint 驱动** | 实施过程不是黑箱，关键节点必须暂停汇报 |
| **证据验收** | 不听"完成了"，要看证据（编译结果、测试结果、diff） |
| **文档即交付** | 改了代码不同步文档 = 没完成 |

---

## 6 阶段标准流程

### Phase 1：反 slop · 目标对齐（Input）

**目的**：消除理解偏差，明确"做什么"和"不做什么"。

**强制动作**：

1. **AI 先复述**：收到任务后，AI 用自己的话复述对目标的理解：
   - 这次要解决什么问题？（新需求 / 修复 / 重构 / 发布）
   - 会触碰哪些模块？影响面？
   - 是否有需要与外部系统保持一致的标识符/契约？
2. **用户纠正**：用户确认正确或纠正偏差
3. **收边界**：明确 scope（做什么）和 anti-scope（不做什么）

**退出条件**：双方对齐一致，无模糊地带。

**标准输出格式**：
```
## 我的理解
- **目标**：<一句话描述>
- **触碰模块**：<列出涉及的模块/文件>
- **做什么**：<scope 清单>
- **不做什么**：<anti-scope>
- **需确认项**：<如有歧义列出>
```

---

### Phase 2：方案设计 + Spec 落地（Plan）

**目的**：将决策从对话固化到文件，作为后续实施和验收的唯一真相源。

**何时必做**：涉及架构调整、新增模块、接口变更、多步骤实施的任务。  
**何时可跳过**：纯机械性单步操作（如改一个配置值）。

**方案对话输出**（至少包含）：
- 改动概述（1-2 句话）
- 关键接口/数据结构/决策点
- 与现有架构的衔接
- 风险与备选

**Spec 落地模板**（强制写入文件）：
```markdown
# <任务标题>

## 核心目标
<1-3 句话>

## 明确不做
<本轮边界>

## 关键决策（含 why）
1. <决策>：<原因>

## 验收维度
- [ ] <可验证的验收标准 1>
- [ ] <可验证的验收标准 2>

## 任务包
- **目标**：本轮交付什么
- **边界**：不能碰什么
- **自由度**：AI 可自行决定的范围
- **checkpoint**：哪些节点必须停下汇报

## 待后续补充
- <已知但本轮不处理的项>
```

**退出条件**：Spec 文件已创建，用户确认方案。

---

### Phase 3：实施 + Checkpoint 推进（Execute）

**目的**：分段推进，避免黑箱实施后大面积返工。

**Checkpoint 机制**（强制）：

AI 必须在以下节点暂停并汇报：
- **动手前**：即将修改的文件、diff 计划、风险点
- **子模块完成后**：进度、下一步、是否发现新问题
- **遇到不确定性时**：主动暂停说明卡点

**Checkpoint 汇报格式**：
```
当前进度：<已完成什么>
下一步计划：<准备做什么>
风险/卡点：<已知风险或不确定性>
```

**用户可执行的 6 种动作**：

| 动作 | 含义 |
|------|------|
| **放行** | 计划合理，继续 |
| **追问** | 先把不确定性搞清楚 |
| **加料** | 沿原方向叠新约束 |
| **绕道** | 方向不对，改方案 |
| **回炉** | 从某个上游节点重来 |
| **阻止** | 先停住 |

---

### Phase 3.5：转向硬规则（Guard Rails）

**目的**：防止 AI 偏离轨道后继续错下去。

触发以下**任一条件**必须立即暂停：

| # | 触发条件 | 动作 |
|---|----------|------|
| 1 | **Spec 冲突**：当前方案违反 spec 中已定的目标/边界/决策 | 暂停，重读 spec，修正方案 |
| 2 | **越界**：改动范围超出任务包允许的边界 | 暂停，向用户确认是否扩大 scope |
| 3 | **连续验证失败**：同一条路径连续 2 次失败且无新证据 | 暂停，向用户汇报，讨论是否换方向 |

**转向后**：回到 spec，以 spec 为准重新规划。转向不是惩罚，是把分叉点说清楚。

---

### Phase 4：多层验收（Verify）

**目的**：用证据而非自我声明来验证完成度。

**第 1 层 · 自验报告**（AI 必须输出）：

| 维度 | 内容 |
|------|------|
| 实际改了什么 | diff 摘要 / 文件清单 |
| 跑了什么验证 | 命令 + 结果 |
| 验收标准逐条对照 | spec 中每条是否覆盖 |
| 没覆盖什么 | 哪些风险/边界未覆盖 |
| 后续排查建议 | 如果出问题从哪里排查 |

**第 2 层 · 自测**：
- 编译/构建通过
- 自动化测试通过（新增 + 回归）

**第 3 层 · 他测建议**：
- 主动建议独立 review 的关注点（3 个）

**第 4 层 · 全量回归**：
- 全量测试命令执行，确认无回归

**退出条件**：4 层验收全部通过，有证据支撑。

---

### Phase 5：文档与进度同步（Sync）

**目的**：改了代码不同步文档 = 没完成。

**通用同步清单**（按项目约定选取适用项）：

- [ ] 架构/设计文档是否需要更新？
- [ ] 用户手册/README 是否需要更新？
- [ ] CHANGELOG 是否需要更新？
- [ ] TODO/Backlog 是否需要双向操作？（完成的删掉，新发现的加上）
- [ ] 进度/里程碑文档是否需要更新？
- [ ] 自动生成的文档是否需要重新生成？

**退出条件**：所有适用的文档已同步。

---

### Phase 6：交付与收尾（Deliver）

**目的**：明确交付物，提示后续动作。

**标准交付汇报格式**：
```
## 交付摘要
- **完成了什么**：<一句话>
- **改动文件**：<文件清单>
- **验证结果**：<编译/测试状态>
- **同步的文档**：<文档清单>
- **Spec 文件**：<路径>

## 后续建议
- <如有后续步骤，主动提示>
```

**收尾动作**（视任务类型选取）：
- Git commit + push（如适用）
- 交接给下游 skill（如 design → verification）
- 通知/分发（如 release）

---

## 通用约束（所有阶段适用）

1. **常量/配置集中管理**：禁止硬编码散落各处的魔法字符串
2. **单一职责**：一个模块/文件只做一件事
3. **不阻断用户**：工具定位为辅助，异常必须捕获降级，不得中断主链路
4. **向后兼容**：调整已有接口/配置时必须在方案中明确告知
5. **外部真相源优先**：当对话记忆与 spec 文件冲突时，以 spec 为准

## 何时停下来问用户（通用触发条件）

- 需求边界存在歧义
- 新改动可能与已有功能冲突
- 需要扩展核心接口/抽象
- 改动影响部署/发布流程
- 触发转向硬规则（spec 冲突 / 越界 / 连续失败）

---

## 如何基于本模板派生特化 Skill

1. **复制本模板**为新 skill 的骨架
2. **填充领域约束**：在每个 Phase 内追加领域特有的规则（如 rule-authoring 在 Phase 3 有"JSDoc 头部必填"）
3. **定制验收标准**：Phase 4 的"跑什么验证"因领域而异（编译/测试/打包/部署）
4. **定制文档同步清单**：Phase 5 的同步项因项目而异
5. **定制收尾动作**：Phase 6 的 commit 格式、分发方式等

> 特化 skill = 通用模板 + 领域约束 + 项目约定
